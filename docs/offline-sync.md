# Offline Sync Flow

How the Caputo Home app handles offline grocery list operations.

## Scope

Only the **Grocery List** section supports offline mode. Freezer, Fridge, Pantry, and Cookbook are online-only since they change less frequently and don't need instant access while shopping.

## Architecture

Three layers work together:

1. **IndexedDB** (`src/lib/db.js`) stores cached grocery items and a write queue
2. **useOfflineSync hook** (`src/hooks/useOfflineSync.js`) manages online/offline state, queuing, and replay
3. **Service Worker** (generated by vite-plugin-pwa / Workbox) caches static assets and Supabase API responses

## IndexedDB Schema

Database name: `caputo-home`, version 1

**grocery_items store** (keyPath: `id`)
- Full mirror of Supabase grocery_items for the household
- Replaced in bulk whenever fresh data arrives from Supabase

**write_queue store** (keyPath: `id`, autoIncrement)
- Queued mutations that couldn't reach Supabase
- Each entry has: `type` (insert/update/delete), relevant payload fields, `timestamp`
- Indexed by `timestamp` for ordered replay

## Sync Sequence

### When online (normal flow)

1. App loads, `useGroceryList` fetches items from Supabase
2. `useOfflineSync.cacheFromSupabase()` writes those items to IndexedDB
3. All writes (add, check, delete, update) go directly to Supabase
4. Supabase Realtime pushes changes back, triggering a refetch
5. Refetched items are re-cached to IndexedDB

### When going offline

1. Browser fires `offline` event, `useOfflineSync` sets `isOnline = false`
2. `OfflineIndicator` component renders a yellow banner
3. Writes are routed through `queueOfflineWrite()` instead of Supabase
4. Each queued write gets a timestamp and goes into the `write_queue` store
5. `pendingCount` updates so the indicator shows "X changes pending"
6. UI reads from the IndexedDB `grocery_items` cache via `getCachedItems()`

### When coming back online

1. Browser fires `online` event, `useOfflineSync` sets `isOnline = true`
2. `syncNow()` fires automatically (triggered by the `isOnline` effect)
3. Pending writes are read from `write_queue` ordered by timestamp (oldest first)
4. Each write replays against Supabase:
   - `insert`: inserts the full item payload into `grocery_items`
   - `update`: patches the item by `itemId` with `changes`
   - `delete`: deletes by `itemId`
5. After each successful write, `clearWrite(id)` removes it from the queue
6. If a write fails, it stays in the queue and sync continues with the next one
7. After all writes processed, `pendingCount` refreshes

### Conflict handling

Last-write-wins. The offline queue replays in chronological order. If both users edited the same item, the last replay overwrites the other. This is acceptable for a 2-person household grocery list.

## Service Worker Caching

Configured in `vite.config.js` via vite-plugin-pwa:

- **Static assets**: All JS, CSS, HTML, images, fonts precached on install
- **Supabase API**: `NetworkFirst` strategy with 24-hour cache fallback
  - Pattern: `https://*.supabase.co/rest/v1/*`
  - Cache name: `supabase-api`
  - Max 50 entries

## Push Notifications

Handled separately from offline sync via `usePushNotifications` hook and `send-push.js` Netlify function. Notifications only fire when the app is online (the function call to `/.netlify/functions/send-push` requires network). Debounced to max 1 push per user per hour.
